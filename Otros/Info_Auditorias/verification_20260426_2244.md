## Detalle por Hallazgo Relevante

### A11Y-11 / SEO-I18N — `html lang` dinámico
- **Severidad original:** HIGH
- **Estado final:** 🔶 REGRESIÓN
- **Qué se mejoró:** `app/[lang]/layout.tsx` ahora renderiza `<html lang={lang}>` y se agregó `app/studio/layout.tsx` para Studio.
- **Nuevo problema introducido:** `app/layout.tsx` quedó sin `<html>` ni `<body>`.
- **Evidencia documental:** la doc oficial de Next.js 16 dice que el root layout **must define `<html>` and `<body>` tags** y que para múltiples root layouts hay que **omitir `app/layout.js`** (`node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/layout.md:114-146`).
- **Conclusión:** el fix va en la dirección correcta conceptualmente, PERO la implementación actual es riesgosa porque contradice un invariante del framework. Esto no es “resuelto”; es “arregló una cosa y abrió otra”.

### BUG-07 — Grid/Lightbox mostrando español en `/en`
- **Severidad original:** HIGH
- **Estado final:** ✅ RESUELTO
- **Evidencia:**
  - `ProductGridCard.tsx` ahora usa `displayName` también en `alt` (`alt={`${displayName} ${product.presentation}`}`)
  - `ProductLightbox.tsx` usa `displayName` y `displayDescription`
  - `ingredients` y `benefits` ya no se muestran en `/en` (`lang === "es" && ...`)
- **Observación:** esto resuelve el síntoma auditado. No convierte mágicamente el catálogo en bilingüe, pero sí elimina el problema visible reportado.

## Evaluación de Madurez de la Skill V2

## Lo que la V2 hizo mejor
1. **Permitió pensar en working tree real**, no solo commits. Eso fue CLAVE otra vez.
2. **Obliga a contrastar respuesta del desarrollador vs evidencia**, y eso mejoró la calidad de la lectura.
3. **El criterio de REGRESIÓN ahora sí sirve**: permitió detectar que el fix de `lang` puede haber introducido un problema framework-level.
4. **La separación entre persistente, parcial y regresión es mucho más madura** que en la v1.

## Lo que todavía le falta para ser realmente madura

### 1. Agregar un paso explícito de verificación con documentación del framework
La V2 todavía no dice algo como:

> “Si un fix toca convenciones del framework, verificar la documentación oficial/versionada antes de marcarlo como resuelto.”

Eso faltó y fue CLAVE en este rerun. Sin leer docs de Next.js, el cambio en layouts podía parecer correcto cuando en realidad contradice una regla fuerte del framework.

### 2. Distinguir mejor entre “síntoma resuelto” y “solución arquitectural correcta”
En algunos hallazgos, como el de `html lang`, el síntoma visual parece arreglado, pero la implementación puede seguir siendo incorrecta a nivel de framework. La skill debería forzar una mini doble validación:
- **nivel síntoma**
- **nivel integridad de implementación**

### 3. Incluir una sección de “riesgos no ejecutables”
Como no se puede hacer build acá por restricción del proyecto, la skill debería tener una salida formal tipo:

```markdown
## Riesgos no verificados por ejecución
- [hallazgo] — [qué no se pudo validar sin build/test/runtime]
```

Eso evitaría sobrerreclamar certezas cuando dependemos de evidencia estática + docs.

### 4. Resolver el problema operativo del loader de skill
Esto no es del texto de la skill, pero SÍ afecta madurez real del flujo.
Si el entorno sigue cargando la v1 mientras el archivo en disco es v2, la skill no es confiable operacionalmente aunque el contenido esté bien escrito.
